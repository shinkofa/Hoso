# Session Report — Hoso — 2026-05-29-001

> Audio Twitch enfin fonctionnel — diagnostic complet + fork StreamPack patché.

## Contexte

Session de continuation sur le bug audio Hoso : la vidéo arrivait sur
Twitch (1920x858 30fps stable) mais l'audio ne sortait jamais.
Twitch Inspector affichait `Audio Track | Codec [vide] | SampleRate [vide]`
malgré des centaines de modifications côté Android. Sessions précédentes
avaient déjà corrigé la couche framework (FGS bitmask, `MediaRecorder.MIC`,
désactivation AEC/NS) sans débloquer Twitch — preuve que le problème
était ailleurs dans la chaîne.

## Objectif L1

Identifier la cause racine du bug audio et la corriger de manière
qualitative et robuste (mots de Jay : « qualitative et robuste »,
pas un workaround).

## Diagnostic L1 → L2 (root cause)

### Couche 1 — Framework Android (OK depuis session précédente)

Logs du device Realme ColorOS confirmaient que la captation OS
fonctionnait :

| Indicateur | Valeur observée |
|------------|-----------------|
| `VD.AudioRecordMonitor isSilenced` | `false` (4 occurrences pendant le stream) |
| `OplusAudioRecordInfoUploadHelper recordTotalTime` | `#67701` (67,7 s capturés) |
| Source | `MIC=1` |
| Format | `PCM_16BIT`, mono, 44 100 Hz |
| Device | `BUILTIN_MIC` |

Conclusion : l'audio sortait bien du mic, traversait l'AudioRecord,
n'était pas silencé par le framework. Le bug n'était pas côté OS.

### Couche 2 — Pipeline StreamPack (cause racine identifiée)

Veille proactive sur GitHub issues du repo `ThibaultBee/StreamPack` +
lecture directe du code source du SDK 3.1.2 (récupéré dans `.research/`).

**Cause racine** : race condition dans
`EncodingPipelineOutput.startStreamUnsafe()`.

Séquence buggée (lignes 595-628 de `EncodingPipelineOutput.kt`) :

```kotlin
// 1. Endpoint configuré (addStreams) mais pas démarré
val streamsIdMap = endpointInternal.addStreams(streams)
audioStreamId = streamsIdMap[audioCodecConfig]

// 2. Listener notifié
streamEventListener?.onStartStream()

// 3. Coroutines encoder lancées EN PARALLÈLE
val audioEncoderJob = audioEncoderInternal?.let {
    coroutineScope.launch { it.startStream() }
}
val videoEncoderJob = videoEncoderInternal?.let {
    coroutineScope.launch { it.startStream() }
}
audioEncoderJob?.join()
videoEncoderJob?.join()

// 4. SEULEMENT MAINTENANT : RTMP publish() handshake
endpointInternal.startStream()
```

Or :

- `audioEncoderInternal.startStream()` fait `MediaCodec.start()` en
  interne. L'encoder AAC émet son **sequence header (csd-0,
  AACPacketType=0)** dans les premières millisecondes — c'est
  l'`AudioSpecificConfig` AAC LC (2 octets pour mono 44 100).
- Un consumer démarré dans `init{}` (lignes 258-272) écoute la sortie
  de l'encoder et appelle `endpointInternal.write(closeableFrame, ...)`
  dès qu'un frame arrive.
- `RtmpEndpoint.write()` (lignes 148-164) **catche silencieusement
  toutes les Throwable** : `catch (t: Throwable) { Logger.w(...) }`.

Conséquence : le csd-0 est écrit AVANT que `RtmpClient.publish()` ait
fini la handshake → write échoue silencieusement (juste une ligne W
dans logcat) → `AudioFlvStream.sentSequenceStart` passe à `true` sur
ce premier frame → **le codec config n'est jamais ré-émis** → Twitch
ne reçoit jamais de descripteur audio AAC → Inspector reste vide à
vie pendant que la vidéo fonctionne normalement.

C'est un bug structurel de StreamPack 3.1.2, pas un problème de
configuration côté Hoso.

## Décision : Fork local + patch

Jay : « J'ai pas envie d'attendre qu'ils valident un PR. Tant qu'à
faire, je préfère qu'on fork et qu'on règle le souci nous-mêmes. »
+ « Je veux que ce soit fait de manière qualitative et robuste. »

Option retenue (option A) : **inverser l'ordre dans
`startStreamUnsafe()`** — déplacer `endpointInternal.startStream()`
AVANT le lancement des coroutines encoder, pas après.

Rationale : c'est la correction la plus structurelle. La handshake
RTMP DOIT être terminée avant qu'un seul frame encodé n'arrive à
l'endpoint. Toute autre solution (verrou, latch, polling) serait un
workaround.
