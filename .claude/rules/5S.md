# 5S — Atelier propre, esprit clair

> Methode Toyota Production System transposee au dev / IA. **Grille de lecture (philosophie), jamais une 7e barriere manuelle** — voir "Statut" en bas.
> Complementaire de Monozukuri : Monozukuri = l'attitude, 5S = les gestes.
> Source : Toyota Production System (Taiichi Ohno) + Lean Software Development (Poppendieck). Transposition Shinkofa 2026-05-19.

## Pourquoi 5S existe

Un atelier en desordre cache les defauts. Un atelier propre rend les defauts evidents au premier coup d'oeil. Le 5S n'est pas du nettoyage cosmetique — c'est un systeme qui force la qualite a etre visible.

Transpose au code : un repo en desordre cache les bugs. Un repo propre fait sortir les anomalies. Les TODO oublies, le code mort, les secrets en clair, les conventions mixees — tout cela cree du bruit dans lequel les vrais problemes se noient.

Trois constats Shinkofa qui justifient le 5S :

1. **Vault coverage 12/24** = traduction directe de Seiri : eliminer le superflu (secrets en clair) prend du temps parce qu'on n'a pas eu la discipline de le faire au moment de la creation.
2. **Blockers cross-projet actifs** (bot-theermite Conflict 409 26K logs/7j, mcp-searxng EADDRINUSE 110K erreurs) = absence de Seiso : le nettoyage n'est pas continu, les erreurs s'accumulent jusqu'au moment ou elles deviennent intenables.
3. **Code drift compose<->runtime** (shizen-shizen vs shinkofa-prod-shizen) = absence de Seiton : chaque chose n'a pas une place unique connue.

## Test de conformite 5S (WARN — fin de session)

A integrer dans le test Monozukuri de fin de session. 6 questions binaires :

1. **S1** — Le repo ne contient aucun code mort, TODO oublie, ou fichier orphelin ajoute cette session ?
2. **S2** — Chaque nouveau fichier est a la place attendue par les conventions (Naming Registry, dossier correct) ?
3. **S3** — Tests verts + lint zero error + encoding UTF-8 propre a la fin de la session ?
4. **S4** — Aucune convention contournee "juste pour cette fois" sans documentation explicite dans le rapport ?
5. **S5** — La discipline de session a tenu jusqu'a la fin (pas de skip de gate non documente) ?
6. **Bonus** — Si une regle a manque, MNK-GoRin a ete enrichie (pas un workaround dans le projet) ?

Une reponse "non" non-justifiee signale un relachement a corriger — un indicateur de lecture, pas une penalite de score automatique. Le 5S reste une grille (philosophie) : il eclaire les regles deja BLOCKING (Quality, Conventions, Workflows), il ne devient jamais une 7e barriere manuelle qui doublerait celles-ci.

## Difference 5S vs Monozukuri

| Axe | Monozukuri | 5S |
|-----|------------|-----|
| Nature | Philosophie — attitude de l'artisan | Operationnel — gestes observables |
| Question centrale | Qui suis-je quand je produis ? | Quel est l'etat de mon atelier ? |
| Test | "Est-ce que je tiens dans 6 mois ?" | "Est-ce que mon poste est propre maintenant ?" |
| Trace | Subjective (commit message, doc) | Objective (lint output, git diff, hooks) |
| Echec | "J'ai cede a la vitesse" | "Le repo contient X TODO et Y secrets en clair" |

Les deux sont necessaires. Monozukuri sans 5S = belle attitude sans evidence. 5S sans Monozukuri = checklist mecanique sans esprit.

## Statut — grille de lecture, jamais une barriere manuelle (decision 2026-06-23)

Le 5S **reste une philosophie / grille de lecture**. Il n'a pas de date de bascule BLOCKING. Pourquoi :

- Les 5 piliers chevauchent partiellement des regles deja BLOCKING (Quality.md, Conventions.md, Workflows.md). Une barriere 5S separee ne ferait que doubler celles-ci.
- Le 5S apporte un **angle de lecture** des regles existantes — sa valeur est de rendre les defauts visibles, pas d'ajouter un controle automatique.
- La bascule BLOCKING prevue le 2026-06-19 reposait sur une calibration de 10 sessions. Constat 2026-06-23 : le test 6 questions n'a jamais ete instrumente (aucun hook, aucune etape `/session-end`), joue 1 fois sur 21 sessions. La calibration est donc invalidable — c'etait un bug d'implementation, pas un signal qualite.
- Decision Jay 2026-06-23 (direction B+C) : le 5S garde son role de grille. Les vrais gardes restent les regles BLOCKING existantes et leurs hooks. Le 5S les eclaire ; il ne les remplace ni ne les double.

Le test de conformite 6 questions reste un **outil de relecture optionnel** en fin de session (signal, pas penalite automatique).

## Pourquoi standalone et pas dans Monozukuri.md

- Monozukuri est philosophique. 5S est operationnel. Fusionner dilue les deux.
- Le 5S sera reference depuis `Quality.md` (gates), `Workflows.md` (session flow), `Conventions.md` (naming/structure). Standalone permet la reference sans alourdir.
- Le 5S a une **histoire transmissible** seule (Toyota Production System). Monozukuri aussi. Garder les deux files-niveau preserve cette transmissibilite.
