# 5S — Atelier propre, esprit clair

> Source complète : github.com/theermite/Shinzo · `07-Methode/Regles/5S.md`
> Méthode Toyota (TPS) transposée au dev. **Grille de lecture (philosophie),
> JAMAIS une 7e barrière manuelle.**

**Niveau** : GUIDE — grille de lecture, pas BLOCKING (voir Statut).

Un atelier en désordre cache les défauts ; un atelier propre les rend évidents.
Transposé : un repo propre fait sortir les anomalies. Le 5S n'ajoute pas de règle —
il organise les règles existantes sous un angle observable.

**Les 5 piliers** :

| # | Pilier | Au dev |
|---|--------|--------|
| S1 | Seiri (trier) | Éliminer code mort, TODO obsolètes, deps inutiles, secrets en clair, branches mortes |
| S2 | Seiton (ranger) | Chaque chose à sa place : structure cohérente, naming, Lego library (zéro duplication), registry à jour |
| S3 | Seiso (nettoyer) | À chaque commit : tests verts, lint zéro, encoding UTF-8 propre, build sans warning |
| S4 | Seiketsu (standardiser) | Une seule façon de faire : templates partagés, hooks code-enforced, conventions automatisées |
| S5 | Shitsuke (maintenir) | La méthode ne s'érode pas : /audit, /session-end, Kata source unique. Code-enforced > human-enforced |

**5S vs Monozukuri** : Monozukuri = l'attitude (qui suis-je quand je produis ?).
5S = les gestes observables (quel est l'état de mon atelier ?). Les deux nécessaires.

**Statut (décision 2026-06-23)** : le 5S reste une grille de lecture, jamais une
barrière manuelle. Les vrais gardes sont les règles BLOCKING existantes (Quality,
Conventions, Workflows) et leurs hooks ; le 5S les éclaire, ne les double pas. Le
test 6 questions de fin de session = outil de relecture optionnel (signal, pas
pénalité automatique).
